# Reflex 研究台重设计 Spec

## 1. 背景

当前 `Streamlit` 里的“关注页”和“主题聚合”能用，但主流程不顺。

现在真实使用顺序更像：

`交易流` -> 看到信号 -> 点进`个股` -> 再看相关`板块`

而不是：

先建关注页 -> 先配关键字 -> 再开始研究

旧设计的问题：

1. 主入口不对，研究前先配置，绕路太多。
2. `AI` 标签不够准，逼着人自己猜关键字、自己搜、自己并。
3. “个股页 / 板块页 / 关注页 / 板块管理”概念混在一起，不够清楚。
4. 旧页更像配置系统，不像研究系统。

这次不是把 `Streamlit` 页面原样搬到 `Reflex`。
这次要做的是：把“研究”放前面，把“归并修正”放后面。

## 2. 目标

### 2.1 要达到的目标

1. 让 `交易流` 继续做主入口。
2. 点`个股标签`或`板块标签`后，自动进入对应研究页，不需要先新建关注页。
3. `个股页`第一屏先看最近买卖信号和原文。
4. `板块页`第一屏先看板块内最近最强信号。
5. 把“归并 / 修正 / 推荐关系”做成辅助层，不再做主流程。
6. 提供一个单独的`整理中心`，集中看待确认关系。
7. 关系确认后，立刻影响`历史数据 + 未来数据`。

### 2.2 明确不做的事

1. 不做“全自动图谱系统”。
2. 不让 `AI` 直接改正式关系。
3. 不保留“关注页”作为新的主概念。
4. 第一版不追求“AI 一次把标签抽完美”。

## 3. 核心产品决定

### 3.1 主入口

主入口仍然是 `交易流`。

在 `交易流` 中：

1. 点整行：看这条信号详情。
2. 点`个股标签`：进入个股研究页。
3. 点`板块标签`：进入板块研究页。

### 3.2 自动研究页

研究页是自动生成的，不需要“新建关注页”。

只要系统能识别到一个研究对象，就可以直接打开它的页。

研究对象分两类：

1. `个股`
2. `板块`

### 3.3 搜索

提供一个全局搜索框。

要求：

1. `个股`和`板块`都能搜。
2. 搜到后直接跳研究页。
3. 搜索是一级入口，不依赖交易流。

### 3.4 板块关系

板块关系不是树，是`网`。

意思是：

1. 一个板块可以和多个板块相关。
2. 一个板块可以同时挂到多个更大方向。
3. `AI` 可以先判断“相关”还是“上下级”，但最后都要人确认。

## 4. 页面信息结构

### 4.1 一级导航

一级导航只保留：

1. `交易流`
2. `整理中心`
3. `全局搜索`

不再把“关注页”放在一级导航。

### 4.2 交易流页

交易流页继续作为研究前门。

核心交互：

1. 顶部保留时间窗、排序、共识筛选等控制项。
2. 主区是信号列表。
3. 列表行内要清楚区分：
   - 信号方向
   - 强度 / 净值
   - 最近动作
   - 原文摘要
   - 个股标签
   - 板块标签
4. 点整行看信号详情。
5. 点标签跳对象页。

### 4.3 个股研究页

个股页第一屏优先看研究内容，不先看整理动作。

推荐结构：

1. 页头
   - 票名
   - 代码
   - 已确认别名
   - 已确认相关板块
2. 第一屏主区
   - 最近买卖信号
   - 原文
3. 第二屏
   - 时间线
   - 提及作者
   - 信号分布
4. 侧边区或下方二级区
   - 待确认关系
   - 可能同票别名
   - 可能相关板块

### 4.4 板块研究页

板块页第一屏先看板块整体动态。

推荐结构：

1. 页头
   - 板块名
   - 已确认相关板块
   - 已确认上下级板块
2. 第一屏主区
   - 板块内最近最强信号
3. 第二屏
   - 板块内个股排行
   - 分歧最大的票
   - 最近共振票
4. 侧边区或下方二级区
   - 待确认关系
   - 可能相关板块
   - 可能上下级板块

### 4.5 整理中心

整理中心是“集中修关系”的地方，不是平时主研究入口。

默认先看：

1. `个股别名 / 代码归并`

后续再看：

2. `个股 -> 板块`
3. `板块 -> 板块`

每条待确认项都要有：

1. 候选对象
2. `AI` 理由
3. 证据摘要
4. 操作按钮：
   - `确认`
   - `忽略`
   - `不再推荐`

## 5. 关系系统设计

### 5.1 关系类型

第一版只做 3 类关系：

1. `个股归并`
   - 某个名字 / 代码 / 标签，是否是同一个票
2. `个股 -> 板块`
   - 某个个股是否属于某些板块
3. `板块 -> 板块`
   - 两个板块是否相关
   - 两个板块是否是上下级

### 5.2 统一处理流程

三类关系都走同一条路：

`规则召回 -> AI 初判 -> 人确认 -> 全局生效`

要求：

1. `AI` 只负责推荐，不直接落正式关系。
2. 人确认后才写正式关系。
3. 正式关系写入后，研究页、整理中心、搜索结果都要更新。

### 5.3 规则召回

第一版只写“最硬、最不容易错”的规则。

#### 个股归并候选

规则示例：

1. 标准化后同股票代码。
2. 同一个名字多次和同一个代码共现。
3. 某个别名长期稳定地和某个个股一起出现。

#### 个股 -> 板块候选

规则示例：

1. 个股和板块词长期共现。
2. 同一批作者反复把该个股和某个方向一起提。
3. 同一时间窗里，经常跟同一类交易信号一起出现。

#### 板块 -> 板块候选

规则示例：

1. 两个板块长期共现。
2. 两个板块的相关个股重合度高。
3. 两个板块的信号节奏长期相近。

### 5.4 AI 初判

`AI` 不从零发明关系。

`AI` 只看规则召回出来的候选，做两件事：

1. 排序
2. 写理由

对于 `板块 -> 板块`，`AI` 可以先判断：

1. 更像“相关”
2. 更像“上下级”

但仍然只作为建议。

### 5.5 人工确认

用户确认后：

1. 关系进入正式关系层。
2. 该关系立刻回灌到历史数据和未来数据。
3. 同时支持撤销。

`忽略`和`不再推荐`必须分开：

1. `忽略`：这次不采纳，以后还可以再出现。
2. `不再推荐`：以后同类候选不要再来打扰。

## 6. 数据层设计

### 6.1 四层结构

底层数据按四层分：

1. `原始层`
   - 现有 `posts`
   - `assertions`
   - `match_keys`
   - `topic_key`
   - `stock_codes`
   - `stock_names`
2. `研究对象层`
   - 个股对象
   - 板块对象
3. `正式关系层`
   - 个股归并关系
   - 个股 -> 板块关系
   - 板块 -> 板块关系
4. `候选推荐层`
   - 待确认项
   - 推荐理由
   - 证据摘要
   - 状态

### 6.2 旧数据如何用

现有数据不丢。

使用原则：

1. `posts` / `assertions` 继续做原始数据源。
2. 现有 `topic_cluster` 相关表可以作为板块初始种子。
3. 现有 `follow_pages` 表不再作为主模型。

`follow_pages` 里的“页面”概念在新系统里退场。
如果里面有仍然有价值的人工配置，只作为迁移参考，不再作为新 UI 的核心概念。
第一版不再提供“按页面维护自由文本关键字”的主流程能力，改为确认离散关系。

### 6.3 生效方式

确认关系后，不重跑整套 `AI` 抽取。

只需要：

1. 写正式关系
2. 重建研究对象视图
3. 让相关页重新按新关系取数

这样可以让生效更快、更稳。

## 7. 视觉和交互方向

视觉上要做成“研究台”，不是“表单后台”。

原则：

1. 主区永远优先展示研究内容。
2. 整理动作放侧边或二级区，不抢主视线。
3. 多用标签、卡片、抽屉，不用一上来就堆大表单。
4. 所有推荐关系都要显示理由，不做黑箱。
5. 风格延续现有 `Reflex` 作业板的暖橙 + 深墨方向，但更克制、更稳。

## 8. 迁移策略

### 8.1 总体策略

不是把旧页原样搬到 `Reflex`。

迁移策略是：

1. 保留并继续增强 `交易流`
2. 新建 `个股研究页`
3. 新建 `板块研究页`
4. 新建 `整理中心`
5. 把旧“关注页 / 板块管理”能力拆成：
   - 后台关系能力
   - 新的整理 UI
6. 等新流程能用后，再退掉旧 `Streamlit` 对应页

### 8.2 分阶段落地

建议分三步：

#### 阶段 1

1. `交易流` 打通跳转
2. 个股页和板块页的静态研究骨架可用
3. 搜索可用

#### 阶段 2

1. 个股归并候选可用
2. 个股 -> 板块候选可用
3. 页内待确认关系可用
4. 整理中心第一版可用

#### 阶段 3

1. 板块 -> 板块推荐可用
2. 历史回灌和重建流程更稳
3. 旧 `Streamlit` 页退场

## 9. 风险和边界

### 9.1 风险

1. 如果规则召回太弱，`AI` 没有足够候选可看。
2. 如果规则召回太宽，待确认项会太多，噪音太大。
3. 如果关系确认后不能快速回灌，用户会觉得系统“不生效”。
4. 如果还保留太多旧“页面配置”思路，最后会变成旧系统换皮。

### 9.2 控制办法

1. 第一版只上最硬规则。
2. 推荐项必须带理由和证据。
3. 所有正式关系支持撤销。
4. 页内待确认只给小批量、高优先级项，避免研究页被整理动作淹没。
5. 整理中心承担批量处理，不把重整理压到研究页。

## 10. 测试和验收

### 10.1 功能验收

至少验证这些场景：

1. 从 `交易流` 点整行能看信号详情。
2. 从 `交易流` 点个股标签能进个股页。
3. 从 `交易流` 点板块标签能进板块页。
4. 全局搜索能找到个股和板块并跳页。
5. 个股页第一屏先看到最近买卖信号和原文。
6. 板块页第一屏先看到板块内最近最强信号。
7. 整理中心默认先看个股归并。
8. 确认一个关系后，历史和未来取数都按新关系生效。

### 10.2 质量验收

1. `AI` 推荐项都有可读理由。
2. 研究页主内容不被整理控件抢走。
3. 不需要“新建关注页”也能完成主研究流程。
4. 旧“关注页”概念不再影响新导航和新页面结构。

## 11. 最终结论

推荐方案是：

`研究优先`

也就是：

1. `交易流` 和 `搜索` 直接进入研究对象
2. 个股页 / 板块页自动生成
3. 关系归并退到辅助层
4. `规则 + AI 推荐 + 人确认` 做关系系统
5. 板块关系采用`网`模型，不采用树模型

这样做，才是真正解决“现在用起来麻烦”的问题，而不是把旧页面换一个前端壳。
